## 1. Executive summary

**The thesis.** P0–P9 (the monorepo consolidation, shipped 8.7) was *physical* consolidation: it absorbed tracer, reporter, and plugin-fees into one repository and one ledger binary, collapsed deploy units, and unified the build. It moved atoms. It did not change what the system *means*. v4 is *semantic* consolidation — identity, flow, and contract. Physical consolidation made the parts live in one place; semantic consolidation makes them speak one language:

- **Identity.** "Who owns this money" must be answerable from the ledger, not inferred across stores. v4 adds `accounts.holder_id`, eager self-holders, and renames the colliding `Alias` CRM entity to `Instrument` so the routing-handle `Account.Alias` and the external-identity entity stop sharing a name.
- **Flow.** Policy enforcement must be inline on the financial hot path, not a side system nobody calls. v4 wires the ledger to the tracer at the transaction-creation seam and converts tracer usage accounting to a two-phase reserve→confirm/release model that provably converges under crash.
- **Contract.** The API surface must be one source of truth. v4 collapses the served swagger so `:3002` enumerates exactly the routes the unified binary mounts, kills the dead `/aliases` surface and `plugin-crm` namespace, and lands a spec-vs-routes diff gate so served and mounted can never silently diverge again.

**The three-ring model.** v4 is organized around three concentric rings plus one orthogonal read-path:

1. **Kernel — double-entry.** The sum-zero invariant. Non-negotiable, untouched by every phase. Every other ring is a guest in the kernel's house; none may perturb balance math.
2. **Policy ring — tracer + fees.** Enforcement that gates transactions: fees already inline at the `transaction_create.go` seam (P-phase work); tracer joins it inline in v4 (D-5/D-6/D-7), sharing the same architectural seam family.
3. **Identity ring — holders + instruments.** Ownership (`accounts.holder_id`, self-holders) and external identity (the renamed `Instrument` entity, banking details). The ring the composition surface (D-9) exposes.
4. **Reporter — orthogonal read-path.** Reporter stays a separate read-side deploy unit. It is not a ring; it consumes the others' data and is touched by v4 only at the contract/version edges (collection-name coupling, version train), never in the write path.

**What ships in v4.0.** The full story, inline, in one train:

- Ownership in the ledger (`accounts.holder_id`, self-holders, backfill, `RequireHolder` opt-in gate) — F1.
- The `Alias→Instrument` hard rename and the `/aliases` contract hard cut — F2.
- Tracer enforcement inline at the transaction seam, two-phase reserve→confirm/release with TTL convergence and per-ledger fail posture — F3.
- The minimal composition surface `POST /v1/holders/:id/accounts` (account + optional instrument in one call) — F4.
- A single harmonized `:3002` contract, version-train aligned to `v4.0.0`, with a CI spec-vs-routes diff gate — F5.
- The Go module bump `/v3 → /v4`, the grep-zero vocabulary sweep, the migration guide, and the release-engineering cut — F6.

Ledger-as-primitive remains the default product. Core-banking mode (RequireHolder, tracer enforce) is configuration, not a fork (D-4, D-5).

```
                  ┌──────► F1 (identity: ownership) ──┐
                  │                                    │
   F0 ───────────┼──────► F2 (instruments: rename) ───┼──► F4 (composition) ──┐
```

Edge rationale (one line each):

- **F0 → F1, F2, F3:** every parallel phase depends on F0's frozen decisions ledger and the green, infra-fixed baseline (gate-zero) before building.
- **F1 → F4:** composition's account write needs F1's `holder_id` column, holder-read port, and self-holder default to populate ownership.
- **F2 → F4:** composition orchestrates the renamed `Instrument` create use case and keys against the unified authz namespace F2 settles.
- **F1, F2, F3, F4 → F5:** the unified spec can only be regenerated once every mounted route (holders/instruments ownership, any tracer HTTP shape change, the composition endpoint) is final; F3 gates F5 specifically.
- **F1..F5 → F6:** F6 is the terminal mechanical major — it bumps the module, sweeps the rename/namespace residue F1–F5 leave behind, and cuts the release; every HEAD-pinned count must be re-measured after the upstream phases merge.

### Design calls made by this plan

- **(d) Branch strategy — RESOLVED (Fred, 2026-06-04): all v4 work continues on `feat/monorepo-consolidation`.** The recommended dedicated `feat/v4` was declined; one branch carries consolidation + v4 history. Consequence accepted: the `/v3→/v4` sweep and v4 breaking history share the branch with consolidation cleanup — so the F0 gate-zero SHA is the **v4 epoch marker**: the gate-zero artifact records it explicitly, and every later bisect/diff range uses it as the v4 base.
- **Capture the baseline after the (c) fixes, not before.** Justification: a pre-fix `make test-integration` with `MIDAZ_TEST_MANAGE_STACK=true` fails at chaos setup on the dead `up-backend` target, and the `:3000/:3001` defaults misroute health checks — a "baseline" taken there records infra rot, not code health. *Rejected alternative:* freeze the baseline at the current HEAD first — produces a red/ambiguous gate-zero that every later phase has to caveat.
- **Inventory all four authz namespaces, not just `plugin-crm`.** Justification: D-10 names `plugin-crm`, but `plugin-fees` is identically external-policy-keyed (`fees_routes.go:16-19`) and `routing` is an under-documented third internal namespace; scoping the register to one namespace guarantees a downstream surprise. *Rejected alternative:* inventory only `plugin-crm` as the charter literally names — narrower, but leaves F-phases blind to the `plugin-fees` policy dependency.
- **Frame D-6 as a redesign of the existing synchronous validate, not a greenfield build.** Justification: `POST /v1/validations` already exists (`routes.go:329`) and already commits usage counters inline-atomically (`validation_service.go:158-166`). Inventorying it as "no endpoint today, net-new on both sides" would send the D-6 implementer down a greenfield path and miss the migration constraints (existing idempotency contract, `usage_counter_repository`, audit-event coupling). F0 records the existing endpoint as the migration surface; the net-new delta is confirm/release + TTL reservation. *Rejected alternative:* describe the validate path as net-new — cleaner narrative, but factually wrong and a trap for the downstream phase.

## 5. Phase F1 — Identity: ownership in the ledger

**Objective.** Make "who owns this money" universally answerable from the ledger without a Mongo round-trip on read. F1 adds a nullable, indexed `accounts.holder_id` column to the onboarding PostgreSQL, materialises it at account-create time (default = the organisation's own self-holder, never null except for `@external/*` accounts), eager-provisions a `LEGAL_PERSON` self-holder when an organisation is created, backfills both surfaces for existing data, and exposes ownership reads (account-list filter + `GET /v1/holders/:id/accounts`). Referential integrity is application-enforced (PG account → Mongo holder is cross-store, no DB FK possible). A per-ledger `RequireHolder` setting (default false) gates a holder-existence check on the create path, mirroring `Accounting.ValidateAccountType` exactly. `holder_id` is immutable in v4.0. With `RequireHolder=false` and no `holderId` supplied, the read and transaction paths are behaviorally equivalent to today; the only F1-introduced changes on the create path are a materialized `holder_id` on every non-external account and one new `holderId` field on the `account.created` wire payload.

**Decisions implemented:** D-2 (ownership link in the ledger: `accounts.holder_id`, rich Holder stays in CRM Mongo, ownership queries ledger-side), D-3 (eager self-holder in `CreateOrganization` + backfill + no-null materialisation, `@external/*` exempt), D-4 (`RequireHolder` per-ledger opt-in following `ValidateAccountType`), D-12 (`holder_id` immutable, transfer = new account + balance-transfer transaction).

### Design calls made by this plan

- **(b) Org→self-holder linkage — CALL: CRM-side deterministic lookup keyed on org ID; NO new organizations-side column, NO flag on holder.** Justification: the self-holder is, by construction, the unique `LEGAL_PERSON` holder in `holders_<orgID>` whose identity derives from the org. Provision it with a deterministic ID derived from the organisation ID (e.g. `holder.ID = uuid.NewSHA1(midazNamespace, orgID.Bytes())`, a UUIDv5), so "the org's self-holder ID" is computable from the org ID alone — no extra column to store, no lookup race, idempotent by primary-key collision on re-provision (the Mongo `_id` IS the holder UUID, `holder.go:17`). *Rejected: organizations-side `self_holder_id` column* — adds an onboarding DDL + write-ordering coupling for a value already derivable. *Rejected: boolean flag on holder* — requires a Mongo `FindAll`+filter to locate the self-holder, defeating the cheap-lookup goal.
- **(b/c) Deterministic-ID write path — CALL: add a `CreateHolderWithID(ctx, orgID, holder)` provisioning method on the CRM holder service that accepts a precomputed deterministic ID, keeping the service+observability boundary intact.** Justification: the existing `CreateHolder` service mints its own UUIDv7 (`components/crm/services/create-holder.go:33` `libCommons.GenerateUUIDv7()`, assigned at `:42`) and discards any caller-supplied ID — so the deterministic-ID scheme from call (b) is infeasible through `CreateHolder`. A dedicated provisioning method satisfies design-call (d)'s "use the CRM service, not the raw repo" principle while allowing the injected ID; PK-collision idempotency on re-provision still holds. *Rejected: call `HolderRepo.Create` directly* — bypasses the service's org→collection mapping and tracing. *Rejected: keep `CreateHolder` and post-update the ID* — Mongo `_id` is immutable; not possible. **Convention break, stated:** holders use time-ordered UUIDv7; a UUIDv5 self-holder sorts unpredictably in `_id`-ascending holder lists. Accepted: the self-holder is an internal record, rarely listed; determinism is worth the ordering wrinkle.
- **(c) Backfill runner — CALL: a standalone idempotent command (not a golang-migrate SQL file), invoked per-tenant.** It (1) iterates orgs from PG `organization`; (2) provisions the deterministic self-holder in `holders_<orgID>` via the new `CreateHolderWithID` path (idempotent — PK collision = already done); (3) `UPDATE account SET holder_id = <self-holder> WHERE holder_id IS NULL AND lower(type) <> 'external'` per org/ledger. Justification: the work spans two stores and cannot live in `NewMigrator` (one PG connection). Re-runs are safe. MT: enumerates tenant DBs and runs inside each tenant's resolved PG+Mongo context. *Rejected: SQL-only migration* — cannot touch Mongo, could materialise `holder_id` pointing at a non-existent self-holder. *Rejected: lazy backfill on first read* — leaves `holder_id` null indefinitely, violating D-3's no-null invariant.
- **(d) Holder-read port location — CALL: `HolderReader` interface defined in the `command` package, satisfied by the CRM holder service (`crmservices.UseCase`), wired in bootstrap. Map zero-rows to a new ledger sentinel + `constant.EntityHolder`.** Justification: dependency-inward — `command` depends on its own narrow interface, not on `components/crm`. The CRM service owns the org→collection mapping and tracing; the port stays minimal (`Exists(ctx, orgID string, id uuid.UUID) (bool, error)`). *Rejected: depend on the raw `holder.Repository`* — leaks the misleadingly-named `collection string` param and the `"holders_"+orgID` mapping into the ledger. *Rejected: import `crmservices` directly in `command`* — couples inner layer to a sibling component.
- **(e) Self-holder lookup caching — CALL: no lookup needed to COMPUTE the default; cache the existence-check result only.** Because the self-holder ID is deterministic from the org ID, the hot path DERIVES the default `holder_id` with zero I/O. The existence *check* (only when `RequireHolder=true`) is gated and lower-frequency. Route the `RequireHolder` settings read through the cached query-side `GetParsedLedgerSettings` rather than the uncached `LedgerRepo.GetSettings` that `applyAccountingValidations` uses (`create_account.go:273`). *Rejected: synchronous Mongo lookup of the self-holder on every account create* — needless round-trip for a derivable value. *Rejected: route the settings read through `LedgerRepo.GetSettings`* — uncached raw DB read; prefer cached query-side reads (Risk R2).
- **(f) Settings grouping — CALL: place `RequireHolder` under the existing `accounting` group as `accounting.requireHolder` (bool).** Justification: it is a create-time validation gate of the same family as `ValidateAccountType`, reuses the existing nested-group deep-merge boundary, and needs no new struct/schema group plumbing. *Rejected: new top-level `holders`/`crm` group* — over-structures a single boolean for no semantic gain in v4.0.
- **(g) Holders-accounts route + authz namespace — CALL: register `GET /v1/holders/:id/accounts` inside `RegisterCRMRoutesToApp` (`components/crm/adapters/http/in/routes.go`), under the existing `plugin-crm` authz namespace, with the handler delegating to the ledger account-query use case via an injected reader.** Justification: the route path is holders-scoped, and every `/v1/holders*` route already lives in the CRM router under `plugin-crm`. A holders-noun route in F1 MUST pick a namespace — it cannot be deferred to D-10 because the route ships in F1. Keeping it under `plugin-crm` is the only choice consistent with the existing holders surface. **Layer-crossing acknowledged:** the handler is registered in the CRM HTTP layer but calls a ledger account query; wire it via a thin reader interface so the CRM router does not import ledger internals. The `plugin-crm` removal (D-10) is a release-train concern that moves/renames this route with the rest of the holders surface; F1 does not pre-empt it. *Rejected: add the route to the ledger `routes.go` under `midaz/ledgers` authz* — splits the holders surface across two routers/namespaces. *Rejected: defer the route entirely to F4* — the charter puts ownership reads in F1.

## 7. Phase F3 — Policy seam: tracer inline with two-phase reservation

**Objective.** Wire the ledger's transaction-creation path to the tracer policy engine inline (D-5), converting tracer usage accounting from a single committed counter into a two-phase reserve→confirm/release model (D-6) whose counters provably converge under crash even when the ledger dies between reserve and balance commit, gated by per-ledger configuration with an explicit fail-open/fail-closed posture (D-7). This is the highest-risk phase of the major: it adds an external dependency to the financial hot path, mutates the tracer's atomic counter invariant that today prevents over-limit, and must do so without touching double-entry correctness. The two-phase model is greenfield in tracer — no `reserve`/`hold`/`pending` concept exists today (`components/tracer/internal/adapters/postgres/usage_counter_repository.go:49-72`, single `current_usage` column; `components/tracer/migrations/000004_initial_schema.up.sql:114-122`).

**Decisions implemented:** D-5 (tracer enforcement inline at the transaction seam), D-6 (two-phase reserve/confirm/release with TTL safety net), D-7 (per-ledger fail posture, default fail-open with SKIPPED audit). Consumes per-ledger settings infra shared with D-4 (RequireHolder), but D-4 itself lands in F1.

### Design calls made by this plan

**A — API shape: new `/v1/reservations` resource, NOT overloading `POST /v1/validations`.**
Call: introduce `POST /v1/reservations` (reserve), `POST /v1/reservations/:id/confirm`, `POST /v1/reservations/:id/release`. Leave `POST /v1/validations` intact as the synchronous one-shot allow/deny path. Justification: `transaction_validations` is DB-enforced append-only (UPDATE/DELETE `DO INSTEAD NOTHING` RULEs, `000004_initial_schema.up.sql:179-197`); a reservation MUST be mutable (reserved→confirmed→released), so it cannot live in the validations table. Overloading `/v1/validations` to "sometimes reserve" would fork the orchestration and break the SOX-immutable read contract on `GET /v1/validations`. Rejected: evolve `POST /v1/validations` to return a reservation + confirm/release sub-resources — conflates an immutable audit record with a mutable lifecycle object.

**B — counter schema: `reserved_usage` column on `usage_counters` PLUS a separate `usage_reservations` table.**
Call: add `reserved_usage BIGINT NOT NULL DEFAULT 0` for the fast in-tx limit check, and a separate `usage_reservations` table holding one row per reservation for per-reservation TTL, idempotent confirm/release, and audit. Reserve increments `reserved_usage` and inserts a reservation row in one tx; confirm moves `amount` from `reserved_usage`→`current_usage`; release/expiry decrements `reserved_usage`. Justification: the column alone reuses the existing atomic CTE with a modified WHERE guard so the hot-path check stays one round-trip, but a single `expires_at` per counter row cannot express per-reservation TTL; the table gives the granularity the reaper needs. Rejected: column-only with one `expires_at` per counter (PENDING and direct reservations have wildly different lifetimes). (Q10-RESOLVED: hybrid accepted.)

DDL sketch:
```sql
ALTER TABLE usage_counters ADD COLUMN reserved_usage BIGINT NOT NULL DEFAULT 0 CHECK (reserved_usage >= 0);

CREATE TABLE usage_reservations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    limit_id UUID NOT NULL REFERENCES limits(id) ON DELETE CASCADE,
    scope_key VARCHAR(255) NOT NULL,
    period_key VARCHAR(50) NOT NULL,
    amount BIGINT NOT NULL CHECK (amount >= 0),
    status VARCHAR(16) NOT NULL DEFAULT 'RESERVED', -- RESERVED|CONFIRMED|RELEASED|EXPIRED
    transaction_id UUID,                            -- ledger correlation, NOT a FK (cross-service)
    reservation_expires_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    confirmed_at TIMESTAMPTZ,
    released_at TIMESTAMPTZ
);
-- Idempotency at the RESERVATION GRAIN. The counter ON CONFLICT key is
-- (limit_id, scope_key, period_key), so a single ledger transaction that touches
-- one limit_id across two scopes/periods yields TWO reservation rows. Keying
-- idempotency on (transaction_id, limit_id) alone would collide on the second row.
CREATE UNIQUE INDEX idx_usage_reservations_request
    ON usage_reservations(transaction_id, limit_id, scope_key, period_key);
CREATE INDEX idx_usage_reservations_reaper ON usage_reservations(reservation_expires_at) WHERE status = 'RESERVED';
```
The new atomic reserve CTE WHERE guard: `current_usage + reserved_usage + $amount <= maxAmount`. Getting this predicate wrong reintroduces the exact TOCTOU over-limit bug the current atomic CTE prevents — the critical correctness point.

**C — ledger settings: new top-level `tracer` group with enum string fields.**
Call: add a `tracer` group to `LedgerSettings` — `tracer.mode` (off|advisory|enforce, default off), `tracer.failPosture` (open|closed, default open), `tracer.timeoutMs` (number, default short e.g. 250). Justification: a top-level group keeps the deep-merge boundary clean and matches `accounting`; storage needs no DDL; read uses the cached `GetParsedLedgerSettings`. Critical: `validateSettingsFieldType` only checks primitive type, NOT enum membership (`settings.go:227-243`) — so `mode`/`failPosture` MUST get explicit allowed-value validation in `ValidateSettings`, or `"enfroce"` persists and silently falls through to default. Rejected: nest under `accounting`.

**D — reserve anchor: immediately before `ProcessBalanceOperations` (`transaction_create.go:1228`), NOT the fee seam slot.**
Call: place the reserve call after the fee-inclusive re-validation (`:1106`) and route validation, immediately before `ProcessBalanceOperations` (`:1228`). Justification: reserve must run on FEE-INCLUSIVE amounts, so it must sit downstream of `applyFees` (`:1075`); and `:1228` is the true balance-commit boundary — synchronous in both modes. Rejected: reserve at the fee-seam slot (`:1075`) — the fee engine mutates `Send.Value` afterward, so a `:1075` reserve would under-count by the fee legs.

**E — async path: reserve at HTTP time, confirm at the synchronous balance commit, NOT in the consumer.**
Call: in async mode the reserve fires at HTTP time (`:1228`), and confirm fires immediately after the synchronous `ProcessBalanceOperations` at `:1228` — NOT in the RabbitMQ consumer. Justification: balances move synchronously at HTTP time in BOTH modes; `RABBITMQ_TRANSACTION_ASYNC` defers only Postgres row persistence. The reserve→confirm window for direct transactions is sub-second. Confirm transport rides the post-commit lib-streaming emit slot. Rejected: confirm in the async consumer — factually wrong (the consumer does not re-mutate balances). One carve-out: legacy `Version==""` payloads DO call `UpdateBalances` in the consumer — transient pre-v3.6 in-flight messages, outside the reserve/confirm seam.

**F — PENDING transactions: reserve at create-pending, confirm at commit, release at cancel; pending-TTL is a deliberate divergence.**
Call: PENDING reserves at `executeCreateTransaction` (status PENDING, `:1130`), confirms at `commitOrCancelTransaction` ProcessBalanceOperations (`transaction_state_handlers.go:525`), releases at cancel. Because PENDING rows persist indefinitely and there is no existing pending sweep, a short reservation TTL would expire reservations for still-valid pendings. Surface options (decide in task expansion): (i) pending reservations get a long/unbounded TTL distinct from direct-transaction TTL; (ii) confirm-on-commit re-reserves if expired; (iii) a dedicated pending-reconciliation worker. Proposal to carry: option (i), per-reservation TTL with pendings flagged long-lived. Rejected: one global TTL.

**G — confirm/release transport: ride the existing post-commit lib-streaming emit slot; durability via the outbox is a prerequisite, not part of this seam.**
Call: confirm/release fire from the post-commit lib-streaming lifecycle-event slot — `emitTransactionLifecycleEvent` (`send_transaction_events.go:173`) → `pkgStreaming.EmitImportant` (`:238`) — through a tracer client, NOT durably-atomic with the DB today. Note the citation split: `SendTransactionEvents` (`:68`) is the LEGACY RabbitMQ publish (`ProducerDefault` `:114`); the durable confirm/release rides the SEPARATE lib-streaming emit at `:173`/`:238`. Justification: the outbox subsystem is NOT wired in midaz; current emits are best-effort. The TTL reaper IS the durability backstop: a lost confirm/release leaves a RESERVED row the reaper releases at expiry, so counters converge regardless of transport loss. This is the architectural reason D-6's TTL is mandatory. Rejected: build the outbox in F3 — scope creep.

**Ledger→tracer client:** new `TracerReserver` port on `TransactionHandler`, injected at `config.go:798` alongside `FeeApplier`, implemented over the `fetcher/client.go` pattern: per-op context timeout from `tracer.timeoutMs`, lib-commons circuit breaker, M2M JWT auth. Breaker wiring is non-trivial: the midaz `CircuitBreakerManager` wrapper is single-service RabbitMQ (only constructor requires a non-nil `*libRabbitmq.RabbitMQConnection`, registers only `rabbitmq.CircuitBreakerServiceName`, wires a RabbitMQ health checker). Anchor the tracer breaker on the fetcher `CircuitBreakerExecutor` pattern — stand up a second breaker registration following that precedent OR generalize the midaz wrapper beyond its RabbitMQ coupling. On timeout/breaker-open, `tracer.failPosture` decides: open → SKIPPED audit + proceed; closed → reject.

**Confirm/release idempotency:** dedup on the reservation-grain unique index `(transaction_id, limit_id, scope_key, period_key)` in `usage_reservations`; a retried confirm/release is a no-op against an already-CONFIRMED/RELEASED row. Reserve dedup carries the ledger `transactionID` so a retried reserve does not double-reserve.

## 8. Phase F4 — Composition surface

**Objective.** Deliver the minimal experience layer: a single authenticated call, `POST /v1/holders/:id/accounts`, that opens a holder-owned ledger account and, when banking details are supplied, an external identity (instrument) in one request. Behavioural contract of `POST /v1/holders/:id/accounts`:
1. Parse holder UUID from path `:id`; parse body (account fields + optional `bankingDetails`/`regulatoryFields`/`relatedParties`).
2. Account creation goes through `CreateAccount` with `holder_id` set to the path holder. Holder-existence is validated inside `CreateAccount` via the F1 holder-read gate. `CreateAlias`/`CreateInstrument` independently re-validates the holder, so the holder is checked before either store is written.
3. Instrument is created ONLY when `bankingDetails` (or other instrument fields) are present (D-8); the composition layer threads the just-created account's `ID` and `LedgerID` into the instrument input. Because the instrument step does NOT verify the account exists, the composition layer is the only thing guaranteeing the threaded `AccountID` points to a real account; in-call this is guaranteed because the account was just created in step 2.
4. Response is a composite `{ account, instrument }`, with `instrument` null when none was requested and an explicit failure marker when the account committed but the instrument write failed.
5. Idempotency: the composition endpoint inherits the account path's idempotency (account is the durable anchor); the instrument step is independently retriable. The composition route does not introduce a second idempotency key.

Partial-failure semantics (explicit, gate-bearing):
- Account commits to PG, instrument write to Mongo fails → the account REMAINS persisted and fully usable. No compensating delete (deleting a balance-bearing ledger account is unacceptable — it is a double-entry primitive).
- The HTTP response returns the created account plus a typed `instrument` failure block (status + reason code).
- Retry path: re-issue `POST /v1/holders/:holder_id/instruments` (the F2 instrument-create route) against the now-existing account, OR re-call composition. No silent server-side compensation. The standalone retry route relies on the caller supplying a correct `AccountID`: there is no server-side account-existence check (Risk 7), so a wrong `AccountID` on retry produces a structurally orphaned instrument rather than a validation error.

### Design calls made by this plan

- **Single atomic Go-import-rewrite commit, separate from the human-doc/config sweep commit.** Call: one mechanical commit does `go.mod:1` + all ~7,116 `.go` import lines (tool-driven: `go mod edit -module` + a path sweep, then `goimports`); a second commit does docs/env/postman/CI. Justification: the Go rewrite is verifiable by `go build ./...` alone and must be reviewable as "nothing but path strings changed"; mixing prose pollutes that diff and breaks bisect. Rejected: one mega-commit — unreviewable, a build failure can't be isolated.
- **Sweep operates on grep-zero of a curated target list, not a blanket `/v3` rewrite.** Call: assert zero hits for each dead token (`mmodel.Alias`, `plugin-crm`, `/aliases`, `:3000`/`:3001`, `LerianStudio/tracer`, residual `midaz/v3`) after excluding gitignored artifacts. Justification: a blanket rewrite corrupts CHANGELOG history and the `Account.Alias` routing handle. Rejected: global `sed s#/v3#/v4#`.
- **The module bump does NOT cut the release tag; a `BREAKING CHANGE` commit on `main` does.** Call: F6 lands the `/v4` bump as code, then a separate commit carrying a `BREAKING CHANGE:` footer (or the `!` marker, e.g. `feat!`) triggers semantic-release to compute `4.0.0` (only valid if the latest tag is `3.x`). Justification: `.releaserc.yml` has no `tagFormat` and reads commit types + latest git tag, never go.mod — and per its `releaseRules` `feat`/`perf`/`build`/`refactor` all map to minor, only `breaking: true` to major. Rejected: assume the go.mod bump (or a bare `feat`) yields a v4 tag.
- **External lockstep is a register, not a repo deliverable.** Call: F6 lists the three external owners (Helm `midaz` chart, `midaz-firmino-gitops`, APIDog scenarios) as a gate to acknowledge before the production tag, owned by P8-T20 / armed-fallback P8-T22. Justification: the in-repo mapping is known but the external repos' structure is unverifiable from here; cutting a real tag before they confirm fans out 4 images into a chart that still keys crm/fees → ArgoCD sync fails. Rejected: cut the tag and fix forward.
- **`up-backend`/`down-backend` repoint is in-scope for F6.** Call: fix `chaos.go:43/49` to call surviving `up`/`down`. Justification: the targets do not exist post-consolidation, so the chaos helper is already broken at HEAD; a one-line repoint. Rejected: leave it as a follow-up.
