## 3. Decisions register (F4.md §5 mandated list + execution)

| Decision | Value | Source / rationale |
|----------|-------|--------------------|
| **`ledgerID` source** | **required `X-Ledger-Id` header** | F4-T06 (`5215c9574`). The composition path `/v1/holders/:id/accounts` is holder-scoped and carries no ledger path segment, so a header source is unavoidable; chosen over a body field to keep the path holder-scoped and mirror how org arrives via `X-Organization-Id`. **Missing header → 400 `ErrMissingFieldsInRequest`; malformed UUID → `ErrInvalidPathParameter`.** No new sentinels introduced — both are reused from `pkg/constant/errors.go` (`composition.go:118`/`:123`). |
| **Swagger `@Tags`** | **`Composition`** | F4-T11 (`5215c9574`). Its own tag group rather than overloading `Accounts`/`Holders` — the composite surface is conceptually distinct. |
| **Authz namespace constant** | **in-package `midazName` constant** + resource **`"accounts"`** | F4-T09 (`5215c9574`). Zero literals, zero `plugin-*`. `midazName = "midaz"` is defined in the ledger `http/in` `routes.go:26` and referenced from `composition_routes.go:26`. **Note:** the CRM tree's `ApplicationName` (`crm .../routes.go:26`) is **same-valued** (`"midaz"`) but a **different-package** constant; the in-package reference was chosen so the composition route keys to the same namespace its sibling ledger routes use, with a single edit point in the ledger tree. |
| **`ParseUUIDPathParameters` locals key** | **`"id"`** (the ACTUAL param name) | Execution finding. The route declares `:id`; `ParseUUIDPathParameters("holder")` is called with `"holder"` as the entityName, but that arg **only labels the span attribute** — it does NOT rename the locals key. The handler reads `http.GetUUIDFromLocals(c, "id")` (`composition.go:72`), keyed on the actual param name, not the entity label. Recorded so a later phase does not "fix" the apparent mismatch by changing the locals key. |
| **Gate-8 swag unblock** | `swaggertype:"string" format:"date"` on `RelatedParty.StartDate`/`EndDate` (`pkg/mmodel/instrument.go:36`/`:38`) | F4-T11 (`5215c9574`). **Doc-only** — it completes the custom `Date` type's declared OpenAPI contract (swag cannot infer the wire shape of a custom type), and matches the committed CRM swagger. No behavior change; required for the isolated ledger leg to generate cleanly. |
| **Business-error assertion idiom** | `errors.As` + `.Code` comparison | Execution finding (test layer, `composition_integration_test.go:473`/`:483`/`:546`). `pkg.ValidateBusinessError` sets the typed error's `.Code` field **without wrapping the sentinel** — so `errors.Is(err, sentinel)` is **false**. Tests must `errors.As` to the typed error and compare `.Code` against `constant.Err*.Error()`. Recorded so the idiom is not mistaken for a bug. |
| **Reason-code extraction** | typed-error `.Code`, with `ErrInternalServer` sentinel **fallback** | F4-T05 (`3eba82331`). `instrumentFailureReason` (`composition.go:149`) walks the typed-error families via `errors.As` and returns the first non-empty `.Code`; an opaque/untyped error falls back to `instrumentFailureFallbackReason = constant.ErrInternalServer.Error()`. Guarantees a stable, client-actionable reason on the `InstrumentFailure` block — never raw internal error text (CLAUDE.md). |
| **Account-field strategy** | **mirror, not embed**, with reflection drift guards | F4-T03 (`96109232e`). `CreateHolderAccountInput` mirrors `CreateAccountInput` field-by-field (HolderID omitted) rather than embedding it. Two guards in `pkg/mmodel/composition_test.go`: **mirror-completeness** (`TestCompositionMirrorsCreateAccountInput` — every account field except `HolderID` is mirrored with matching type/json/validate tags; F4-local 1) and **no-holder-field** (`TestCompositionHasNoHolderIDField` — the composite must not declare `HolderID`; pins the no-new-null-semantics contract). |
| **Partial-failure injection** | `InstrumentCreator` **port stub** against a **REAL committed PG account** | F4-T12 (`41855ccbf`). `erroringInstrumentCreator` satisfies the `InstrumentCreator` port and always errors — failing the instrument leg deterministically **after** the account commits, without touching the account leg or corrupting Mongo. Proves R23 (no compensating delete) at the service boundary with a real surviving account row + a successful standalone retry. |
| **R7 isolation proof shape** | drives the **REAL** `compositionTenantMiddleware` over HTTP with **two real per-tenant managers** | F4-T14 (`41855ccbf`). A context-seam-only test would not catch the precise leak R7 names (generic-Mongo-key bleed onto the PG-keyed onboarding context); only the real middleware with real per-tenant managers exercises key resolution under concurrency. `errgroup` + `-race`. |
| **MT vs single-tenant test split** | single-tenant **static-Mongo** for the behavior suite; MT isolation isolated to T14 | F4-T12/T14 (`41855ccbf`). The behavior suite (Gates 1–4, 6) uses a static single-tenant Mongo connection — MT isolation is **T14's** sole job, kept separate so a behavior failure and an isolation failure never mask each other. |
| **`RequireHolder`/`ValidateAccountType`** | left **default-off** in tests | Execution finding. Both left at their defaults in the composition test setup to keep the suite focused on the composition seam (orchestration, D-8, partial-failure, isolation), not on inherited gates those flags govern. |

## 5. Inheritance flags for F5

1. **The composite response reuses `account.created` unchanged — no new event.** Design call g (deferred) holds: composition adds NO streaming event and reuses `account.created` as emitted by the inherited `CreateAccount`. R48 (emit-signature drift) is not bitten by F4 because F4 adds no event. **If F5 introduces an `instrument.created` event, the drift-trio obligation (`pkg/streaming/events/<event>.go` Payload + constructor + JSONShape test) lands there, not retroactively in F4.** The `HolderAccountResponse` composite is an HTTP response, not a wire payload — no streaming contract attaches to it.

2. **Instrument-step account-existence non-guarantee (R39) — documented, benign in-flow.** The CRM instrument-create use case validates **holder** existence only; it does NOT validate **account** existence. Inside the composition flow this is benign because the account was just created in-call and its ID is threaded through. But the standalone `POST /v1/holders/:holder_id/instruments` route carries the same non-guarantee: a wrong `AccountID` on a standalone call orphans. **F4 added no server-side account-existence check** (out of scope; pre-existing CRM behavior, F2's call). F5/migration docs should carry this non-guarantee forward.

3. **The standalone-retry contract is the documented recovery path for partial failure.** On instrument-write failure, F4 surfaces a typed `InstrumentError` block and the account survives — F4 added NO server-side retry mechanism. The recovery contract is: **re-issue the standalone `POST /v1/holders/:holder_id/instruments` against the surviving account** (proven retriable by `TestIntegration_CompositionPartialFailureNoCompensation`), OR re-call composition. This belongs in the F5/F6 **migration guide** as the client-driven recovery procedure.
