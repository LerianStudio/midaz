# F4 — Composition surface (task breakdown)

> **Status:** task breakdown of `docs/v4/PLAN.md` §8 (Phase F4). The macro plan is authoritative for scope, gates, and risks; this file decomposes them into atomic, independently-verifiable tasks. Where a task contradicts the macro plan, the macro plan wins and the divergence is recorded in the F4 execution note.
>
> - **Date:** 2026-06-04
> - **Branch:** `feat/monorepo-consolidation` (single-branch v4 per Q1)
> - **Phase deps (DAG §3):** needs F1 (`accounts.holder_id`, holder gate inside `CreateAccount`, self-holder) **and** F2 (Alias→Instrument rename, instrument-create use case, unified authz namespace). F4 gates F5.

F4 composes existing primitives behind one new route; it reimplements nothing. Double-entry, holder-existence, and instrument validation are inherited unchanged. The hard, novel work is exactly two things: (1) the cross-store tenant middleware that resolves onboarding-PG **and** CRM-Mongo in one request without key collision (R7), and (2) the non-compensating partial-failure contract (R23). Everything else is wiring that mirrors the proven CRM/fees registrar pattern.

**HEAD-vs-execution note.** F4 sits downstream of F1+F2. Several touchpoints below reference symbols F1/F2 introduce or rename (`mmodel.CreateInstrumentInput`, `CreateInstrument`, the F2 unified authz namespace, `mmodel.Account.HolderID`, `CreateAccountInput.HolderID`). At THIS HEAD those are still `mmodel.CreateAliasInput` / `CreateAlias` / `plugin-crm` / no-`HolderID`. Every such reference is written in its **post-F1/F2 form**; the verified-at-HEAD form is noted inline so an implementer running before the merges knows what to expect. The composition tasks must NOT be merged before F1+F2 land (hard DAG edge).

### F4-T01 — Cross-store composition tenant middleware + two-tenant isolation (HARD GATE, first task of the phase)

**Objective.** Build a dedicated `compositionTenantMiddleware` that injects BOTH the onboarding-PG tenant DB (module-keyed) AND the CRM-Mongo tenant DB (generic key) into one request context, attached ONLY to composition routes, so the generic Mongo key never reaches ledger routes and one tenant's Mongo never leaks into another's request. This is R7 — the highest-risk item in F4 and the phase's first task per the charter.

**Touchpoints (verified at HEAD).**
- `components/ledger/internal/bootstrap/config.go:1180-1186` — the existing onboarding/transaction `tmmiddleware.NewTenantMiddleware(WithPG(onboardingPGManager, constant.ModuleOnboarding), …, WithMB(onboardingMongoManager, ModuleOnboarding), …)`.
- `config.go:1207-1210` — the CRM-only middleware: bare `WithMB(crmMongoManager)` (NO module name — the CRM repos read `tmcore.GetMBContext` on the GENERIC key; the comment at `:1190-1206` documents exactly why).
- `config.go:1229-1233` — the fees-only middleware (same bare-`WithMB` rationale).
- `config.go:23` — `tmmiddleware "github.com/LerianStudio/lib-commons/v5/commons/tenant-manager/middleware"`.
- `buildUnifiedRouteSetup(...)` signature at `config.go:1138-1149` already receives `onboardingPGManager`, `crmMongoManager`, `tenantCache`, `tenantLoader` — all inputs the composition middleware needs; NO new manager parameter required.

**Change shape.** Inside `buildUnifiedRouteSetup`, after the `feesTenantMiddleware` block (`:1229-1233`), add a fourth `NewTenantMiddleware` instance:
```go
compositionTenantMiddleware := tmmiddleware.NewTenantMiddleware(
    tmmiddleware.WithPG(onboardingPGManager, constant.ModuleOnboarding),
    tmmiddleware.WithMB(crmMongoManager), // bare: CRM repos read the generic MB key
    tmmiddleware.WithTenantCache(tenantCache),
    tmmiddleware.WithTenantLoader(tenantLoader),
)
```
Module-keyed `WithPG(...ModuleOnboarding)` matches what the onboarding account repo resolves; bare `WithMB(crmMongoManager)` matches the generic key the CRM instrument repo reads. Add a doc comment mirroring the CRM block's isolation rationale (attached ONLY to composition routes; never global; never on ledger routes). ~12 LOC + comment. The transaction-PG manager is intentionally excluded (composition only writes the onboarding account + the CRM instrument; it never touches the transaction PG).

**Verification (narrowest first).**
1. `go build ./components/ledger/internal/bootstrap/` — compiles with the new middleware.
2. `grep -n "compositionTenantMiddleware" components/ledger/internal/bootstrap/config.go` — shows the instance built.
3. Isolation proof is the dedicated integration test in **F4-T14** (the hard gate); this task lands the middleware, T14 proves isolation under concurrency.

**Dependencies.** F1 (the onboarding account write path with `holder_id`), F2 (CRM instrument repo/manager naming). Inter-phase: hard edge F1+F2 → F4 (§3 DAG).

**Notes.** Do NOT add the composition middleware to the existing `tenantMiddleware` (`:1180`) or call `f.Use(...)` globally — both collide the generic Mongo key with the module-keyed onboarding injection on ledger routes (the precise leak R7 names). The single-tenant path (`!cfg.MultiTenantEnabled` early return at `:1153`) leaves `compositionRouteOptions` nil, same as the other options — the registrar must tolerate nil routeOptions (the CRM/fees registrars already do).

**Dependencies.** F1 (CreateAccount with holder_id), F2 (CreateInstrument), F4-T03.

**Notes.** Span lifecycle per CLAUDE.md: `defer span.End()` after `tracer.Start`; do NOT overwrite parent `ctx` for in-memory mapping. The orchestrator depends on the two use-case surfaces, not on adapters (Design call a; respects handlers→services→adapters).

### F4-T05 — Non-compensating partial-failure semantics in the service

**Objective.** Codify R23: when the account commits to PG but the instrument write to Mongo fails, the account REMAINS persisted and usable, NO compensating delete fires, and the failure is surfaced as a typed `InstrumentError` block for client-driven retry.

**Touchpoints.**
- The `CreateHolderAccount` method in `components/ledger/internal/services/composition/` (F4-T04, step 4).
- `pkg/mmodel/composition.go` `InstrumentFailure` type (F4-T03).

**Change shape.** In step 4 of `CreateHolderAccount`: wrap the `CreateInstrument` call so that on error, the method does NOT return an error and does NOT delete the account — instead it logs a Warn (`libLog.Err(err)`, structured, no PII), span-records the error (`libOpentelemetry.HandleSpanError`), and populates `resp.InstrumentError = &mmodel.InstrumentFailure{Status: "FAILED", Reason: <reason code>}`. The HTTP layer then returns 201 (account succeeded) with the failure block, NOT a 5xx. ~20-30 LOC inside the existing method.

The reason code must be a stable, client-actionable string (e.g. the instrument error's `constant` code), never the raw internal error text (CLAUDE.md: do not expose internal technical error details). No financial values, balances, or PII in the Warn log.

**Verification.**
1. `go build ./components/ledger/internal/services/composition/`.
2. `grep -n "InstrumentError\|HandleSpanError\|LevelWarn" components/ledger/internal/services/composition/*.go` — confirms the no-fail-on-instrument-error path and the structured Warn.
3. Behavioural proof is F4-T12's partial-failure injection test (Gate 4).

**Dependencies.** F4-T04.

**Notes.** Deleting a balance-bearing ledger account is unacceptable (it is a double-entry primitive) — the gate is the ABSENCE of a compensating delete, proven by F4-T12. The retry contract (re-issue the standalone `POST /v1/holders/:holder_id/instruments` against the surviving account, OR re-call composition) is documentation, captured in the F5/F6 migration notes — F4 does not add a server-side retry mechanism.
