### F1-T10 — cross-store backfill runner

**Objective.** A standalone idempotent command that provisions deterministic self-holders for existing orgs (Mongo first) then materialises `account.holder_id` for non-external accounts (PG), per tenant. NOT a SQL migration.

**Touchpoints (HEAD-verified).**
- Orgs in PG `organization` table; accounts in PG `account` (`000005_create_account_table.up.sql`).
- Holders in Mongo per-org `holders_<orgID>` (`holder.mongodb.go:110`).
- MT resolution: Mongo via `tmcore.GetMBContext`, PG via `tmcore.GetPGContext` (per-tenant contexts).

**Change shape.** A new command-style runner (mirror the existing migration/worker invocation pattern, NOT `libPostgres.NewMigrator` — that holds one PG conn and cannot touch Mongo). Per tenant DB: (1) iterate orgs from PG `organization`; (2) provision the deterministic self-holder via `CreateHolderWithID` (idempotent — PK collision = done); (3) `UPDATE account SET holder_id = <derived self-holder> WHERE holder_id IS NULL AND lower(type) <> 'external'` per org/ledger. Re-runs are safe. Medium-to-large.

**Verification.**
- Integration: seed orgs + accounts (incl. one `@external/*`) → run backfill twice → diff PG `account.holder_id` and Mongo `holders_<orgID>` counts run1 vs run2 → identical (idempotency, gate 4). Assert external accounts retain `holder_id IS NULL`; self-holder count == org count.
- MT integration: two tenants with same-shape org UUIDs → backfill on tenant A leaves tenant B untouched (gate 5).

**Dependencies.** F1-T06 (provisioning), F1-T08 (derivation scheme shared). Gate 4, gate 5.

**Notes.** Ordering is mandatory: Mongo self-holder first, then PG materialisation — so `holder_id` never points at a non-existent holder (R15). `@external/*` exempt (D-3 no-null applies only to non-external). The runner is `@external`-exempt and idempotent; it doubles as the repair path for any F1-T09 miss (R34).
