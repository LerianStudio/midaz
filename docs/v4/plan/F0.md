### F0-T04 — Normalize build tags across all non-unit suites (unit stays UNTAGGED)

**Objective.** Make every non-unit test suite reachable by a tag-gated target and consistently tagged, while keeping unit tests UNTAGGED so the bare `go test ./...` invocation keeps running them (tagging unit would make the bare run execute zero tests). This is the structural fix that lets a root `make ci` (F0-T05) cover the full matrix, including `./tests/reporter`.

**Touchpoints (HEAD, verified).**
- `mk/tests.mk:234` and `:329` — `find ./components ./pkg -name '*_integration_test.go'`. The glob excludes `./tests`, so `tests/reporter/integration/*` (under `./tests`) is unreachable by `test-integration`/`coverage-integration`.
- `mk/tests.mk:109` — `test-unit` discovers via `go list ./... | awk '!/\/tests($|\/)/' | awk '!/\/api($|\/)/'` with `go test -race` and NO `-tags`. Confirmed untagged (verified: no `-tags` on the `:124`/`:115`/`:118` invocations).
- Build tags present in `tests/` (verified): `integration`, `chaos`, `e2e`, `fuzz`, `property`, plus the combined `integration || chaos`.
- `tests/reporter/` suite tag distribution (verified): integration=16, property=9, chaos=15, e2e=26 files; all already carry a `//go:build <tag>` line.
- `mk/tests.mk:293` (`test-bdd`, `-tags e2e`, tracer-specific, needs a running service); `mk/tests.mk:223` (`test-integration`, `-tags=integration`). The `property` and `chaos` tags have NO dedicated make target (verified: no `-tags=property`/`-tags=chaos` anywhere).

**Change shape.** Two-part normalization in `mk/tests.mk` (and any new sibling target it warrants):
1. Make non-unit suites discoverable beyond `./components ./pkg` — widen the integration/coverage discovery to also reach `./tests` (or add a dedicated reporter-suite target), so `tests/reporter/integration/*` is no longer orphaned. Per §4 the systemic fix is tag-driven, not glob-widening alone; the chosen mechanism is settled at task expansion but MUST make `property` and `chaos` suites invokable by a target (today they are not).
2. Audit every non-unit suite for a correct `//go:build` tag; add missing tags so each suite is gated by exactly one of `integration|chaos|e2e|fuzz|property` (the combined `integration || chaos` form is permitted where a chaos test lives in an integration file). Unit suites get NO tag — leave them bare.

Mechanically: this is Makefile/recipe surgery plus, where a suite file is mistagged or untagged, a `//go:build` header edit. No production Go code changes.

**Verification.**
```

# Every non-unit suite carries exactly one build-tag line:

```

**Dependencies.** F0-T02, F0-T03 (the harness should orchestrate a stack/env that the fixed helpers can drive). Inter-phase: F0 owns the harness; F6 Gates 8/13 consume `make ci` (built in F0-T05).

**Notes.** Q16-RESOLVED (Fred 2026-06-04). Mitigates R25. This is the harness-authoring task; F0-T05 is the orchestration entrypoint. The drift class this kills: targets and suites silently diverging — after this, a suite without a target (or vice versa) is a gate failure, not invisible.
